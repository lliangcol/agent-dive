# Ruflo problem troubleshooting record

## Problem: Claude Code plugin lite is confused with CLI full install

- Discovery time: 2026-06-11
- Current status: README level confirmed, operation not verified
- Scope of impact: Claude Code plugin, Ruflo CLI init, MCP server, hooks

### Phenomenon

The README clearly separates the installation path into Claude Code plugin and CLI install. The plugin path only provides lightweight capabilities and does not register the complete Ruflo MCP server; CLI install only enters the full loop.

### Preliminary judgment

- Possible reasons: Both paths are called Ruflo, and the warehouse contains a large number of plugin and CLI files.
- Risk: Users think that after installing a single plugin, they will have complete MCP tools such as `memory_store`, `swarm_init`, `agent_spawn`, etc.

### Subsequent processing

- Record file writing and available commands for plugin lite and CLI full install respectively.
- Do not overlay installation paths; if overlay is required, make a rollback plan first.

## Problem: hooks fail and are not blocked

- Discovery time: 2026-06-11
- Current status: source code level confirmed, operation not verified
- Scope of influence: Claude Code hook-backed scenario

### Phenomenon

The hook command of `.claude-plugin/hooks/hooks.json` usually ends with ` |  | true`. Comments indicate that the shim will try to be no-op and not allow CLI/install failure to block Claude Code turn.

### Preliminary judgment

- This helps reduce the impact of hook failures on development.
- But it also means that these hooks cannot be used directly to force-block security access.

### Subsequent processing

- Separately test stdout/stderr/exit code for security, quality, and configuration protection hooks.
- Strong security requirements should be placed in CI or a gate that supports hard blocking.

## Problem: Windows hooks are different from POSIX hooks

- Discovery time: 2026-06-11
- Current status: source code level confirmed, operation not verified
- Scope of impact: Windows, PowerShell, Claude Code hooks

### Phenomenon

The `.claude-plugin/hooks/hooks.json` annotation indicates that the file is POSIX-only, using `/bin/bash`, `jq`, `xargs`, `.sh` scripts. The node-based equivalent configuration is written by `ruflo init` under Windows.

### Preliminary judgment

- Windows behavior cannot be directly inferred using POSIX hooks.json in the repository.
- Windows requires separate verification of the `.claude/settings.json` generated by init.

### Subsequent processing

- Run the init wizard in a Windows temporary project.
- Record the generated hook command and script path.
- Trigger a low-risk hook to do smoke.

## Problem: MCP stdio stdout pollution

- Scope of influence: MCP server, JSON-RPC client

### Phenomenon

### Preliminary judgment

- The source code has been protected, but it still needs to be authentic `mcp start` smoke.

### Subsequent processing

- Use MCP client to request `tools/list` and confirm that stdout only has JSON-RPC.
- Test parse error, unknown method and large payload behavior.

## Problem: CLI init has a large writing range

- Discovery time: 2026-06-11
- Current status: README level confirmed, operation not verified
- Scope of influence: project directory, user Claude/Codex configuration, MCP config

### Phenomenon

README clearly states that CLI install will write `.claude/`, `.claude-flow/`, `CLAUDE.md`, helpers, settings, and register MCP server and hooks.

### Preliminary judgment

- Executing directly in the main repository may pollute the existing Agent configuration.
- Need to verify in temporary empty project first.

### Subsequent processing

```bash
npx --yes ruflo@latest init --help
npx --yes ruflo@latest init wizard
```

The second item can only be executed in temporary projects, and all new/modified files will be recorded.
